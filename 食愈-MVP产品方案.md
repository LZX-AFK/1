## 一、产品概述

### 1.1 产品名称

| 名称类型 | 名称 | 说明 |
|---------|------|------|
| 小程序名称 | 团团食记 | 微信小程序前台显示名称 |
| IP 名称 | 小熊猫团团 | 主宠物角色名 |

### 1.2 产品简介

《团团食记》是一款饮食记录微信小程序，以小熊猫团团为核心陪伴角色，通过**拍照识别 + 角色对话 + 收集解锁 + 小叶能量**的组合，让用户在无压力、无焦虑的氛围中轻松记录每一餐。

**v1.4.0 升级定位：** 首页体验从数据工具型首页升级为**宠物陪伴型主界面**。用户打开小程序后首先看到的是团团的主场景，而不是数据仪表盘。饮食数据以底部圆角面板承载，团团是第一主角，数据是辅助信息。

> 「打开就是见到团团，记录只是顺手完成。」

### 1.3 核心价值主张

| 价值 | 说明 |
|------|------|
| 宠物主场景 | 团团不是装饰，而是首页主角 |
| 记录即奖励 | 每次记录都有结束感和正反馈 |
| 陪伴不监督 | 团团是搭子，不是教练或监督者 |
| 无焦虑设计 | 不道德化评价、不催促、不羞辱、不制造负罪感 |
| 轻量养成 | 小叶能量 + 角色收集，增强留存但不制造亏欠 |
| 流程有陪伴 | 拍照、扫描、识别、保存、失败、空状态都有团团陪伴 |
| 状态有反馈 | 每一次记录都通过团团状态、小叶、对话完成情绪闭环 |
| 首页即 Hub | 主要功能都能从首页轻松进入 |

### 1.4 目标用户

| 维度 | 描述 |
|------|------|
| 年龄 | 18–28 岁 |
| 性别 | 以女性为主 |
| 特征 | 在意身材但不极端节食，生活节奏快，习惯用手机记录日常 |
| 动机 | 想变健康但不想太麻烦，喜欢有成就感的养成类产品 |
| 场景 | 每次点餐或吃饭前后拍照记录，晚上看今日总结 |

### 1.5 竞品差异化

| 维度 | 传统热量记录工具 | BitePal | 团团食记 |
|------|----------------|---------|---------|
| 记录方式 | 手动检索输入 | 拍照识别 | 拍照识别 |
| 宠物机制 | 无 | 浣熊陪伴 + 喂养/health points + 装扮/挑战/奖励 | 小熊猫团团 + 小叶能量（无惩罚） |
| 反馈风格 | 数据导向 | 营养分数 + 即时反应 | 温和陪伴 + 角色对话 |
| 焦虑程度 | 高（数字焦虑） | 中（宠物掉血压力） | 低（无惩罚、无亏欠） |
| 首页体验 | 数据仪表盘 | 宠物场景 + 数据面板 | 宠物主场景 + 底部数据面板 |

## 二、小熊猫团团 IP 规范

### 2.1 角色设定

| 维度 | 内容 |
|------|------|
| 名称 | 团团（TuanTuan） |
| 物种 | 原创小熊猫 / 红熊猫 |
| 性格 | 温柔治愈、好奇嘴馋、偶尔俏皮 |
| 定位 | 饮食记录搭子、食物衣柜管理员 |
| 说话风格 | 第一人称"团团"，语气温暖自然，不刻意卖萌 |

### 2.2 视觉特征

- 暖橘棕色毛色（主色 `#D4845A`）
- 奶油白脸颊和腹部（`#FFF5EB`）
- 圆眼睛，黑色瞳孔带白色高光
- 蓬松环纹大尾巴
- **头顶小叶芽装饰**：团团的标志性装饰，象征新生与成长

### 2.3 六状态定义

| 状态 | 使用场景 | 文案示例 | 动画 |
|------|---------|---------|------|
| standard | 首页默认、拍照页、欢迎页、权限说明页 | "今天也慢慢来，我们从一餐开始。" | tuan-breathe |
| happy | 今日有记录、保存成功、点击团团反馈、回首页更新状态 | "记录下来就已经很好啦。" | tuan-bounce-tail |
| waiting | 首页空状态、今日无记录、衣柜空状态、日历空状态、统计数据不足 | "想开始的时候，我都在这儿陪你。" | tuan-slow-breathe |
| analyzing | 上传中、扫描准备、扫描中、保存中、加载中 | "我来帮你看看这一餐里都有什么。" | tuan-float |
| confused | 识别失败、网络错误、权限错误、删除确认、保存失败 | "没关系，我们再试一次。" | tuan-tilt |
| unlock | 小叶解锁、角色卡解锁、新朋友出现 | "新朋友来啦，一起去衣柜看看吧。" | tuan-unlock-pop |

## 三、团团小叶能量机制

### 3.1 机制目标

| 目标 | 说明 |
|------|------|
| 让每次记录都有结束感 | 记录完成 → 获得 +1 小叶 → 即时正反馈 |
| 让用户跨天回来有理由 | 本周陪伴进度、小叶累计解锁 |
| 轻量累计奖励增强留存 | 达到阈值触发轻解锁，不复杂 |
| 不制造饮食焦虑和宠物亏欠感 | 不扣除、不掉血、不惩罚 |

### 3.2 基础规则

| 规则 | 说明 |
|------|------|
| 每成功记录一餐 | +1 小叶 |
| 当天成功记录 ≥ 1 餐 | 计入本周陪伴 1 天 |
| 当天成功记录 ≥ 2 餐 | 可点亮今日陪伴徽章（非 MVP-A 阻塞项） |
| 累计 3 / 7 / 15 片小叶 | 触发轻量解锁奖励 |
| 不记录不扣除 | 无惩罚机制 |
| 偏油 / 偏甜 / 丰盛餐不扣除 | 所有记录都被团团温柔接住 |

### 3.3 本周陪伴规则

| 规则 | 说明 |
|------|------|
| 本周陪伴 | 当天成功记录 ≥ 1 餐即算 1 个陪伴日 |
| weeklyActiveDates | 记录最近 7 天内有记录的自然日日期（string[]） |
| weeklyActiveDays | weeklyActiveDates.length，用于展示"本周陪伴 X/7 天" |
| 今日陪伴徽章 | 当天成功记录 ≥ 2 餐可点亮（非 MVP-A 阻塞项） |
| 不记录不扣除 | 不显示断签惩罚 |

### 3.4 奖励设计（MVP 阶段只做轻奖励）

**做：**

| 奖励类型 | 说明 |
|---------|------|
| 小叶数量展示 | 首页显示累计小叶数 |
| 保存成功后的团团反馈卡 | 记录成功时团团 happy + 对话 |
| 达到阈值时的轻解锁弹窗 | 3/7/15 片小叶时触发 |
| 日历页展示本周陪伴进度 | 7 天内记录天数 |
| 可解锁简单贴纸 / 背景 / 小印章 | 文案轻量虚拟奖励 |

**不做：**

| 不做项 | 原因 |
|--------|------|
| 小窝家具系统 | 复杂度过高 |
| 装扮商城 | 需要完整经济系统 |
| 虚拟货币消费 | MVP 先验证核心价值 |
| 复杂任务系统 | 增加开发负担 |
| 强制分享 streak | 不给用户压力 |

### 3.5 小叶能量边界说明

- **小叶不是货币：** 不用于购买、不用于投喂团团
- **小叶只是记录留下的温和证明：** 服务于陪伴感，不制造任务压力
- **小叶中心：** 可作为轻量奖励展示页，但不做商城
- **首页展示原则：** 服务于陪伴感，不制造任务压力

### 3.6 与 BitePal 机制对比

| 维度 | BitePal | 团团食记 |
|------|---------|---------|
| 宠物状态 | 浣熊陪伴 + 喂养/health points 叙事 | 只有状态图（无数值） |
| 不记录后果 | 可能存在宠物亏欠感或 health points 叙事 | 无后果，团团安静等待 |
| 饮食评判 | 营养分数、健康提示 | 不道德化评价，只提供温和结构性反馈 |
| 正反馈 | 喂养成功、连胜 | 小叶 +1、团团 happy |
| 用户感受 | 部分用户有 guilt | 无亏欠感 |

## 四、首页产品定义（v1.4.0 重写）

### 4.1 首页定位

- **首页名称：** 团团宠物主场景首页
- **首页路径：** `pages/index/index`
- **首页定位：** 首页是《团团食记》的主场景和全局入口。用户打开小程序后，首先看到团团在温暖的小叶森林场景中等待或回应用户。今日记录、热量、营养占比、小叶、本周陪伴、拍一餐入口都围绕团团组织。

> 「打开就是见到团团，记录只是顺手完成。」

### 4.2 首页必须包含的元素

| 元素 | 说明 |
|------|------|
| 团团主视觉区 | 团团在场景中的状态图，占据首页视觉中心 |
| 团团对话气泡 | 根据状态显示对应文案 |
| 小叶计数 | 显示累计小叶数 |
| 本周陪伴进度 | 显示本周陪伴 X/7 天 |
| 今日已记录餐数 | 显示今日记录数量 |
| 底部今日数据面板 | 热量进度、花费进度、营养占比（可展开/收起） |
| 最近一餐 | 显示最近一次记录的缩略信息 |
| 拍一餐主按钮 | 主要 CTA，醒目位置 |
| 快捷入口 | 食物衣柜、饮食日历、我的、晚餐推荐 |

### 4.3 首页状态定义

| 状态 | 触发条件 | 团团状态 | 文案 |
|------|---------|---------|------|
| empty | 新用户首次进入，无任何记录 | waiting | "想开始的时候，我都在这儿陪你。" |
| waiting | 老用户今日尚未记录 | waiting | "今天也慢慢来，我们从一餐开始。" |
| active | 今日已有至少 1 条记录 | happy | "记录下来就已经很好啦。" |
| afterSave | 刚完成一次记录 | happy | "这一餐被团团认真收下啦。" |

### 4.4 底部数据面板

- **收起态（默认）：** 只显示简要数据
- **展开态：** 用户上滑或点击展开，显示完整数据和列表

### 4.5 首页导航策略

| 策略 | 说明 |
|------|------|
| 首页是主 Hub | MVP 阶段首页是主入口 |
| 主 CTA | "拍一餐"按钮 |
| 快捷入口 | 食物衣柜、饮食日历、我的、晚餐推荐 |
| 不做 tabBar | 默认采用首页 Hub + 页面跳转 |

### 4.6 首页禁止事项

| 禁止 | 说明 |
|------|------|
| 爱心血条 | 不做宠物生命值显示 |
| 宠物生命值 | 不做 health points |
| 饥饿值 | 不做饥饿度系统 |
| 投喂按钮 | 不做投喂交互 |
| Store | 不做商城入口 |
| Plus | 不做付费会员 |
| 付费入口 | 不做任何付费引导 |
| 排行榜 | 不做用户排名 |
| PK | 不做竞争功能 |
| 强制打卡压力 | 不做打卡提醒和惩罚 |

## 五、完整用户流程（v1.4.0 重写）

### 5.1 首次进入流程

```
启动页（Logo + 加载动画）
    ↓
欢迎页（介绍团团，温馨欢迎文案）
    ↓
无压力原则说明（产品不制造焦虑、不评判饮食）
    ↓
健康免责声明（营养数据仅供参考，不构成医疗建议）
    ↓
基础资料设置（可跳过：昵称、年龄、身高、体重、目标）
    ↓
相机权限说明（需要相机权限用于拍照记录）
    ↓
首页宠物主场景（团团 waiting 状态）
```

### 5.2 首次记录流程

```
首页空状态（团团 waiting）
    ↓ 点击「拍一餐」
拍照/相册选择
    ↓
图片预览确认
    ↓
上传图片（团团 analyzing）
    ↓
AI 扫描中（团团 analyzing → 扫描动画）
    ↓
识别结果页 → 展示识别结果卡片
    ↓
用户编辑或确认（可手动修改食物名/热量/份量，可选填价格）
    ↓ 点击「记下来」
保存中（团团 analyzing）
    ↓
保存成功（团团 happy + 成功反馈）
    ↓
+1 小叶反馈（Toast）
    ↓
若累计小叶达到阈值（3/7/15）→ 小叶解锁弹窗（团团 unlock）
若首次记录该食物类别 → 角色卡解锁弹窗（团团 unlock）
    ↓
返回首页有记录状态（团团 happy）
```

### 5.3 老用户回访流程

```
首页宠物主场景（团团 happy 或 waiting）
    ↓ 查看今日记录/小叶/本周陪伴
    ↓ 点击「拍一餐」
拍照/相册选择 → 图片预览确认 → 上传 → AI扫描 → 识别结果
    ↓ 编辑或确认 → 点击「记下来」
保存成功 → +1 小叶反馈
    ↓ 若触发解锁 → 弹窗
返回首页状态更新
```

### 5.4 识别失败流程

```
扫描中
    ↓
识别失败（团团 confused + 失败提示）
    ↓ 用户选择：
    ├── 重新选择图片 → 返回拍照页
    └── 手动记录 → 手动输入食物名称和热量 → 保存成功 → +1 小叶
```

**识别失败场景：**

| 场景 | 团团状态 | 文案 |
|------|---------|------|
| 图片模糊 | confused | "这次没看清，我们换张图试试。" |
| 无法识别食物 | confused | "团团不太确定这是什么，手动记一下也可以哦。" |
| 网络超时 | confused | "网络有点慢，我们再试一次。" |
| AI 服务异常 | confused | "暂时无法识别，手动记一下也可以。" |

### 5.5 数据不足统计流程

```
用户进入统计/趋势页
    ↓ 判断记录天数
    ├── < 3 天 → 显示数据不足提示（团团 waiting）
    │    文案："再记录 X 天，就能看到更多饮食趋势啦。"
    └── ≥ 3 天 → 显示趋势图表 + 数据摘要
```

**数据不足提示对照：**

| 记录天数 | 提示文案 |
|---------|---------|
| 0 天 | "开始记录后，就能看到饮食趋势啦。" |
| 1 天 | "再记录 2 天，就能看到更多饮食趋势啦。" |
| 2 天 | "再记录 1 天，就能看到更多饮食趋势啦。" |
| ≥ 3 天 | 显示趋势图表和数据摘要 |

**数据不足提示原则：** 不说"数据不足无法分析"、不说"你还没有坚持够"、不制造失败感、不使用警告颜色、不强制打卡，只做温和引导。

## 七、核心功能矩阵

### F1 拍照识别 + 记录卡片（P0，核心链路，MVP-A）

**功能说明：** 拍照识别和今日汇总是「记录」大功能的两个子环节，F1 产出卡片，加入记录后首页自动更新。

#### 完整链路

```
用户点「拍一餐」
    ↓
wx.chooseMedia → 拿到图片
    ↓
调百度智能云菜品识别 API
    ↓
返回：{ food_name, calorie_per_100g, nutrition_ratios }
（脂肪/蛋白质/碳水占比由百度返回，热量按份量换算）
    ↓
[并行触发]
├── 豆包图像生成 API → 生成该食物的 chibi 皮肤图片
└── 豆包文本 API → 生成该食物角色的拟人对话文案
    ↓
渲染「识别结果卡片」展示给用户
    ↓
用户填写价格（手动输入，可选填）+ 可手动编辑食物名/热量
→ 点「记下来」
    ↓
写入 meals 表 + 获得 +1 小叶 + 触发衣柜解锁检测（异步）
    ↓
返回首页，今日热量/花费/小叶数实时更新
```

#### 识别结果卡片结构

```
┌──────────────────────────────────┐
│   [食物 chibi 皮肤图片]（豆包生图）│
│   芝士汉堡                        │
│   ────────────────────────────── │
│   热量：640 kcal                  │
│   脂肪 ████░░  38%                │
│   蛋白质 ███░░  27%               │
│   碳水 █████░  35%                │
│   ────────────────────────────── │
│   💬「哈哈我来啦！虽然我油脂有点   │
│   多，但偶尔吃一次也没事哦～记得    │
│   今晚轻食补回来！」（豆包拟人文案）│
│   ────────────────────────────── │
│   价格：¥ [用户填写，可选]        │
│   [记下来]  [重新拍照]             │
└──────────────────────────────────┘
```

#### 各字段数据来源

| 字段 | 来源 |
|------|------|
| 食物名称 | 百度智能云菜品识别 |
| 热量（kcal） | 百度返回 calorie_per_100g × 估算份量（用户可选大/中/小份） |
| 脂肪/蛋白质/碳水占比 | 百度菜品识别营养数据 |
| chibi 皮肤图片 | 豆包图像生成 |
| 拟人对话文案 | 豆包文本生成 |
| 价格 | 用户手动填写（可不填，默认"-"） |

#### Prompt 模板

**豆包生图：**
```
Cute chibi {food_name} character,
kawaii Japanese style, soft green and warm palette,
white background, rounded sticker style, no text
```

**豆包拟人对话：**
```
你是{food_name}的拟人角色，性格活泼可爱。
这个食物热量{calories}kcal，脂肪占比{fat}%，蛋白质{protein}%，碳水{carb}%。
用第一人称说一句角色台词（40字以内），
内容要贴合这个食物的营养特点，语气俏皮，带一个小建议。
```

#### 边界情况

| 场景 | 处理方式 |
|------|---------|
| 百度识别不到 | 提示「没认出来，帮我填一下食物名？」→ 手动输入后继续生成皮肤和对话 |
| 生图/生文耗时 3-8 秒 | 团团 analyzing 状态 + loading 动画占位，不阻塞 |
| 生图失败 | 用食物类别默认占位图（预备 10 张按类别区分） |

### F3 AI 晚餐推荐（P1，MVP-B）

#### 入口和触发时机

```
两个入口：
① 首页「今晚推荐」卡片区域（自动加载）

```

```
豆包返回：推荐 recipe_id 列表 + 每条的推荐理由（1 句话）
    ↓

```

#### TDEE 自动计算

```
BMR（基础代谢）= 哈里斯-本尼迪克特公式
    女：655 + (9.6×体重) + (1.8×身高) - (4.7×年龄)
    男：66 + (13.7×体重) + (5×身高) - (6.8×年龄)

TDEE（每日总消耗）= BMR × 活动系数
    久坐 × 1.2 / 轻度 × 1.375 / 中度 × 1.55 / 高强度 × 1.725

热量目标
    减脂 = TDEE - 300 kcal
    增肌 = TDEE + 200 kcal
    维持 = TDEE
```

系统计算后展示给用户，用户可手动覆盖为自定义值。

## 十五、统计与数据不足机制（v1.4.0 新增）

### 15.1 数据不足提示机制

| 记录天数 | 提示文案 |

| 2 天 | "再记录 1 天，就能看到更多饮食趋势啦。" |
| ≥ 3 天 | 显示趋势图表和数据摘要 |

### 15.2 温和引导原则

| 原则 | 说明 |
|------|------|
| 不说"数据不足无法分析" | 使用温和引导 |
| 不说"你还没有坚持够" | 不制造失败感 |
| 不显示失败感 | 不使用警告颜色 |
| 不强制打卡 | 只做温和引导 |

## 十七、健康合规免责声明

### 17.1 营养数据声明

- **仅供参考：** 本应用提供的营养数据（热量、蛋白质、脂肪、碳水化合物等）均由 AI 识别估算，仅供参考，不构成医疗建议。
- **不替代专业意见：** 如有特殊饮食需求或健康问题，请咨询专业营养师或医生。
- **数据误差：** AI 识别可能存在误差，实际营养含量以食品包装或专业检测为准。

### 17.2 健康提示

- **不制造焦虑：** 本应用不评判用户的饮食选择，不制造饮食焦虑。
- **不强制打卡：** 用户可自由选择记录频率，不强制每日打卡。
- **不替代医疗：** 本应用不是医疗工具，不能替代专业诊断和治疗。

### 17.3 隐私保护

- **数据安全：** 用户数据仅用于本应用功能，不会出售或分享给第三方。
- **图片处理：** 用户上传的图片仅用于食物识别，识别完成后保留作为记录凭证。
- **用户控制：** 用户可随时删除自己的记录数据。

### 17.4 UI 合规设计原则

- 不出现"超重"、"肥胖"、"不达标"、"减肥失败"等标签
- 热量进度条用区间颜色而非超标警告
- 推荐功能标注「仅供参考」
- 所有饮食建议用温和语气，不用命令式表达
- 不显示 Store、Plus、会员、付费等商业入口
- 不显示生命值、血条、饥饿度、投喂等亏欠感系统
