## 一、产品概述

**食愈**是一款面向年轻女性的 AI 饮食记录助手，主打「拍照即记录、游戏化养成」的核心体验。

用户拍一张食物照片，百度智能云菜品识别 API 自动识别食物名称与营养数据，豆包 API 为每种食物生成专属 chibi 皮肤图片和拟人对话卡片；系统根据当日摄入从内置食谱总纲中智能推荐晚餐方案（分居家烹饪和外卖两种）；每记录一种新类别食物，解锁对应专属角色卡，构成「食物衣柜」收藏系统。

**核心价值主张**


| 痛点        | 解决方案               |
| --------- | ------------------ |
| 手动记录饮食太麻烦 | 拍照 → AI 识别，3 秒完成记录 |
| 减脂计划坚持不下去 | 游戏化收集角色卡，把记录变成收集   |
| 不知道今天该吃什么 | 根据当日饮食从食谱总纲推荐晚餐    |
| 饮食和花费难以追踪 | 自动统计每日热量，价格用户自填    |

## 二、目标用户

**核心用户画像**

- **年龄**：18–28 岁
- **性别**：以女性为主
- **特征**：在意身材但不极端节食，生活节奏快，习惯用手机记录日常
- **动机**：想瘦但不想太麻烦，喜欢有成就感的养成类产品
- **场景**：每次点餐或吃饭前后拍照记录，晚上看今日总结

**用户行为路径**

```
吃饭前/后拍照 → 看到热量卡片 + 角色拟人对话 → 参考晚餐推荐 → 解锁新角色卡 → 第二天继续记录
```

### F1 — 拍照识别与记录卡片（P0）

F1 与 F2 属于「记录」大功能，F1 负责产出卡片，F2 负责加入后的首页汇总更新。

#### 完整链路

```
用户点「拍照记录」
    ↓
wx.chooseMedia → 拿到图片临时路径
    ↓
图片转 base64 → 调云函数 identify
    ↓
百度智能云菜品识别 API 分析
    ↓
返回：{ food_name, calorie_per_100g, fat_ratio, protein_ratio, carb_ratio, category }
    ↓
[并行触发，不阻塞用户]
├── 豆包图像生成 API → 生成该食物 chibi 皮肤图片
└── 豆包文本 API → 生成该食物角色拟人对话文案
    ↓
渲染「识别结果卡片」展示给用户
    ↓
用户填写价格（手动输入，可不填，默认显示「-」）
    ↓
点「加入今日记录」
    ↓
调 addMeal → 写入 CloudDB，食物图片上传云存储
    ↓
异步触发 checkAndUnlockSkin（不阻塞主流程）
    ↓
返回首页，今日热量 / 花费实时更新
```

#### 识别结果卡片结构

```
┌──────────────────────────────────┐
│                                  │
│      [食物 chibi 皮肤图片]        │
│         （豆包生图）              │
│                                  │
│   芝士汉堡              640 kcal  │
│   ────────────────────────────── │
│   脂肪   ████████░░░  38%         │
│   蛋白质 ██████░░░░░  27%         │
│   碳水   █████████░░  35%         │
│   ────────────────────────────── │
│   💬「哈哈我来啦！虽然我油脂有点  │
│   多，但偶尔吃一次也没事哦～      │
│   今晚轻食补回来！」              │
│            （豆包拟人对话）        │
│   ────────────────────────────── │
│   价格 ¥ [____]（用户填写）       │
│                                  │
│   [加入今日记录]    [重新拍照]    │
└──────────────────────────────────┘
```

#### 各字段数据来源


| 字段              | 来源                           |
| --------------- | ---------------------------- |
| 食物名称            | 百度智能云菜品识别                    |
| 热量（kcal）        | 百度返回 calorie_per_100g × 估算份量 |
| 脂肪 / 蛋白质 / 碳水占比 | 百度菜品识别营养数据                   |
| chibi 皮肤图片      | 豆包图像生成 API                   |
| 拟人对话文案          | 豆包文本生成 API                   |
| 价格              | 用户手动填写                       |
| 食物原始照片          | 用户拍摄，上传云存储                   |


#### 豆包生图 Prompt 模板

```
Cute chibi {food_name} character sticker,
kawaii Japanese style, soft green and warm palette,
white background, rounded card design, no text
```

#### 豆包拟人对话 Prompt 模板

```
你是{food_name}的拟人角色，性格活泼可爱。
这个食物热量{calories}kcal，脂肪{fat}%，蛋白质{protein}%，碳水{carb}%。
用第一人称说一句角色台词（40字以内），
贴合该食物的营养特点，语气俏皮，带一个轻松的小建议。
```

#### 边界情况处理


| 情况              | 处理方式                               |
| --------------- | ---------------------------------- |
| 百度识别不到食物        | 提示「没认出来，帮我填一下食物名？」→ 手动输入后继续生成皮肤和对话 |
| 生图 / 生文耗时 3-8 秒 | loading 动画占位，图片加载完成前先显示文字信息        |
| 生图失败            | 用食物大类默认占位图（预备 10 张按类别区分）           |
| 网络超时            | 自动重试一次，仍失败则提示用户检查网络                |

### F2 — 今日饮食汇总（P0）

#### 链路

```
首页 onLoad / 从识别结果页返回时
    ↓
调 getTodayMeals
    ↓
返回：{ meals[], total_calories, total_price }
    ↓
热量进度条 = total_calories / users.calorie_target（系统计算值）
花费进度条 = total_price / users.budget
餐次列表渲染，每条显示：食物缩略图 + 名称 + 热量 + 价格 + 时间 + 删除按钮
```

**依赖**：用户已完成资料设置；若未设置则使用默认值（热量目标 1500kcal，预算 ¥50）

### F3 — AI 晚餐推荐（P1）

#### 入口与触发条件

- **入口一**：首页「今晚推荐」卡片区域（自动加载）
- **入口二**：底部 TabBar「食谱」Tab（主动进入）
- **触发条件**：当天有至少 1 条饮食记录；否则显示「先记录今日饮食，我来帮你推荐今晚吃什么」

#### 完整数据链路

```
用户进入食谱页 / 首页推荐区域加载
    ↓
读取今日 meals：{ food_list, total_calories, nutrition_summary }
读取 users：{ goal, budget, calorie_target, activity_level, dietary_preference }
    ↓
计算剩余空间：
    remaining_calories = calorie_target - total_calories
    remaining_budget   = budget - total_price
    ↓
规则预筛食谱总纲：
    剩余热量 < 300kcal → 只看 calories ≤ 400 的食谱
    剩余预算 < ¥20    → 过滤 price 超出的
    饮食偏好          → 排除过敏 / 不吃的食物类别
    ↓
把预筛后候选食谱列表（约 10-15 条）+ 今日饮食摘要喂给豆包
Prompt：「用户今日已摄入{total}kcal，吃了{food_list}，
目标是{goal}，还有{remaining}kcal空间，
从以下食谱中选 2-3 个最合适的，
返回 JSON：[{ recipe_id, reason }]」
    ↓
按 recipe_id 查食谱总纲，拼装为推荐卡片渲染
```

#### 推荐列表页效果（对应 UI 图⑤）

```

[居家] 番茄牛肉汤

  推荐理由：今日蛋白质摄入不足，牛肉补充优质蛋白

[外卖] 轻食沙拉拼盘
  360kcal  ¥35  美团

[居家] 低糖水果酸奶碗
  280kcal  ¥12  ⏱5分钟   ★☆☆
  推荐理由：热量余量有限，轻盈甜品满足口欲
```

### F5 — 食物衣柜（P2）

#### 核心设计

衣柜按**食物大类**区分（20 类），而非按具体食物名，避免卡片无限增多。每个大类首次记录时解锁唯一专属角色卡。

| 类别 key    | 角色名  | 触发示例食物            |
| --------- | ---- | ----------------- |
| staple    | 主食小人 | 米饭 / 面包 / 馒头      |
| burger    | 汉堡小人 | 汉堡 / 三明治 / 热狗     |
| noodle    | 面条小人 | 拉面 / 意面 / 米线      |
| meat      | 肉肉小人 | 鸡肉 / 牛肉 / 猪肉 / 排骨 |
| seafood   | 海鲜小人 | 鱼 / 虾 / 蟹 / 贝类    |
| egg       | 蛋蛋小人 | 鸡蛋 / 茶叶蛋 / 溏心蛋    |
| tofu      | 豆腐小人 | 豆腐 / 豆浆 / 豆皮      |
| vegetable | 蔬菜小人 | 各类蔬菜              |
| fruit     | 水果小人 | 各类水果              |
| soup      | 汤汤小人 | 各类汤品              |
| congee    | 粥粥小人 | 各类粥 / 稀饭          |
| pizza     | 披萨小人 | 披萨 / 意式食物         |
| sushi     | 寿司小人 | 寿司 / 日料           |
| hotpot    | 火锅小人 | 火锅 / 麻辣烫          |

| other     | 神秘小人 | 识别不到类别时兜底         |


#### 解锁完整链路

```
用户点「加入今日记录」→ addMeal 写入成功
    ↓
异步调 checkAndUnlockSkin({ openid, category: "burger" })
    ↓
查询 skins 表：该用户有没有 category="burger" 的记录
    ↓
[已有] → 返回 { is_new: false }，前端不做提示
    ↓
[没有] → 查预生成图片库，取 burger 对应 URL
         写入 skins 表
         返回 { is_new: true, card_image_url, name: "汉堡小人" }
    ↓
前端收到 is_new=true
    ↓
弹出解锁动画：卡片从中心放大淡入 + 撒彩带特效
展示：「汉堡小人已解锁！」+ 卡片预览
用户点击进入衣柜或点 × 关闭
```

**实时生图触发条件**（仅用于 20 类之外的边缘情况）：
百度返回的 category 不在预定 20 类中 → 调 generateSkin（豆包生图）→ 生成后存云存储 + 写入 skins 表，约 5-10 秒，后台静默完成，完成后衣柜页自动刷新。

#### 系统自动计算热量目标（TDEE）

```
BMR（基础代谢）— 哈里斯-本尼迪克特公式
    女：655 + (9.6 × 体重) + (1.8 × 身高) - (4.7 × 年龄)
    男：66  + (13.7 × 体重) + (5 × 身高) - (6.8 × 年龄)

TDEE = BMR × 活动系数
    久坐      × 1.2
    轻度运动  × 1.375
    中度运动  × 1.55
    高强度    × 1.725

热量目标
    减脂 = TDEE - 300 kcal
    增肌 = TDEE + 200 kcal
    维持 = TDEE
```

系统计算后展示给用户，用户可手动覆盖为自定义值。

```
用户打开小程序
```

### Git 分支策略

```
main          只放能完整运行的代码，每天晚上合并一次
feat/A-xxx    A 的开发分支
feat/B-xxx    B 的开发分支
feat/C-xxx    C 的开发分支
```

**日常工作流**

```bash
# 每天开始工作
git checkout main && git pull
git checkout -b feat/A-result-card

# 写完提交
git add miniprogram/pages/result/
git commit -m "feat: 识别结果卡片UI还原"
git push origin feat/A-result-card

# GitHub 发 Pull Request → 另一个人简单 review → merge 到 main
```

**提交信息规范**

```
feat:     新功能
fix:      修 Bug
ui:       纯样式调整
refactor: 重构，不改功能
data:     食谱总纲数据录入
```
