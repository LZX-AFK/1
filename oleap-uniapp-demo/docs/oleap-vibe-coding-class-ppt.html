// 01

为什么这门课要这样做

学生不只是学习某个框架，而是在学习把真实设备能力变成业务产品的路径。

真实输入

不用假数据糊弄自己

录音来自 Oleap 耳机，App 必须完成扫描、连接、录音和文件输出。真实链路通了，后面的业务才有意义。

工程拆分

前端和后端各司其职

uni-app 负责采集和展示，Fastify 负责接收、处理和保存。学生会第一次看见完整应用系统的形状。

AI 加速

Vibe Coding 不是代写

AI 可以帮忙生成页面、接口和提示词，但学生必须能运行、验证、解释和修正。

// 02

Vibe Coding 工作法：让 AI 参与，但不让 AI 接管

学生开始动手前，先建立一个可验证的协作节奏：任务切小、上下文说清、运行验证、错误反馈。

1
明确目标
先说选题、用户、页面和接口，不直接要完整项目。
2
给上下文
说明两个项目、已有 filePath、当前要改哪个文件。
3
改一小块
一次只做一个按钮、一个接口、一个页面状态。
4
运行验证
AI 写完不是结束，能跑通才算完成。
5
反馈错误
把报错、文件路径、刚改内容发回给 AI 继续迭代。
上下文

先告诉 AI 你在哪个项目里

例如：oleap-uniapp-demo 是手机端，oleap-fastify-demo 是后端，前端已经能拿到 audio.filePath。

边界

明确不能改哪里

例如：不要修改 uni_modules/oleap-ble-sdk/utssdk，不要重写整个项目，只改当前页面或当前接口。

目标

一次只给一个小目标

例如：只增加“调用后端处理”按钮，只新增 /api/knowledge/create 接口。

位置

要求 AI 说明修改文件

让 AI 说清楚改哪个文件、哪个方法、哪个页面区块，避免生成一堆不知道放哪的代码。

验收

给出成功标准

例如：保留 onLoad(query) 读取 filePath，显示 loading/error，展示 summary、keywords、suggestions。

反馈

把报错原样发回去

不要只说“不能运行”。要贴出报错、相关代码、刚才修改了什么，请 AI 给最小修复方案。

不推荐：目标太大
容易让 AI 重写、跑偏、破坏现有链路
展开 
帮我做一个语音知识库系统，代码全写好。
推荐：小步、清晰、可验收
适合课堂二次开发
展开 
我正在基于 oleap-uniapp-demo 做二次开发。
当前只改 pages/transcript/transcript.vue。
已有 filePath 来自 onLoad(query)。

请增加一个按钮：调用 Fastify 后端 /api/ai/process。
要求：
1. 不修改 SDK
2. 保留 filePath 读取逻辑
3. 增加 loading/error 状态
4. 展示 summary、keywords、suggestions
5. 给出 template 片段和 methods 代码
课堂口径：Vibe Coding 的关键不是“会不会写神奇提示词”，而是能否把任务切小、给足上下文、运行验证，并把错误反馈成下一轮可执行修改。
// 03

BLE 最小扫盲：知道 App 和耳机怎么说话

今天不要求学生写 BLE 底层，只要能理解主流程和常见故障发生在哪一层。

1
scan 扫描
手机在附近寻找 Oleap 耳机。
2
connect 连接
找到设备后建立通信关系。
3
write 发命令
App 发开始录音、停止录音、查询电量。
4
notify 收数据
耳机主动推送录音帧和状态变化。
5
filePath 结果
SDK 输出 WAV/MP3 路径，业务从这里开始。
知道就好

service discovery

连接成功后，手机还要发现耳机有哪些功能通道，例如控制通道和录音通道。所以“已连接”不等于“录音马上可用”。

知道就好

characteristic

可以把它理解成具体的读写/通知通道。SDK 会缓存这些通道，页面不用自己处理。

知道就好

MTU

MTU 可以粗略理解为 BLE 一次能装多少字节。它影响传输效率，但本项目里由 SDK 处理。

课堂口径：学生今天主要调用 SDK API，不直接操作 BLE 底层。SDK 已经处理服务发现、特征缓存、notify 订阅、MTU、音频帧落盘和解码。

双项目架构

课堂上，VSCode workspace 里建议放两个兄弟项目。

1
Oleap 耳机
提供真实录音、Flash 文件和设备状态。
2
uni-app
扫描连接、开始录音、停止录音。
3

业务链路最关键的数据入口。
4
Fastify
接收请求或上传文件，返回处理结果。
5
业务页面
展示转写、摘要、翻译、标签、建议。

  • 不要一上来改 uni_modules/oleap-ble-sdk/utssdk

// 07

下午第一段：Fastify 后端速成

后端不是额外知识点，它是录音进入业务系统的入口。

启动顺序

先查 IP，再启动服务

cd oleap-fastify-demo
npm install
npm run check
npm run ip
npm run dev
课堂口令：手机访问后端，写电脑 IP，不写 localhost。先用手机浏览器打开 /health，能打开再写 uni-app 请求。
关键接口
  • GET /health 检查服务
  • POST /api/ai/process mock 业务处理
  • POST /api/audio/upload 上传音频文件
  • GET /api/records 查看本地记录
// 08

下午第二段：App 调后端

先用 filePath 字符串跑通，再做真实上传。步子小一点，成功率高很多。

第一步

uni.request：确认 App 能调后端

const API_BASE_URL = 'http://192.168.2.118:3000'

uni.request({
  url: `${API_BASE_URL}/api/ai/process`,
  method: 'POST',
  data: {
    topic: '语音知识库系统',
    filePath: this.filePath
  },
  success: (res) => {
    const result = res.data.data
    this.summary = result.summary
    this.keywords = result.keywords
  }
})
第二步

uni.uploadFile：上传真实录音文件

uni.uploadFile({
  url: `${API_BASE_URL}/api/audio/upload`,
  filePath: this.filePath,
  name: 'audio',
  formData: {
    topic: '语音知识库系统',
    scene: 'classroom'
  },
  success: (res) => {
    const body = JSON.parse(res.data)
    this.summary = body.data.result.summary
  }
})

排错地图

课堂现场最省时间的方式：先判断问题卡在哪一层。

设备层

扫描不到或连不上

  • 检查蓝牙和权限
  • 确认耳机在附近且可连接
  • 重启 App 或蓝牙
网络层

手机访问不了后端

  • 同一 WiFi 或手机热点
  • 不用 localhost，用电脑 IP
  • Windows 防火墙允许 Node.js
接口层

请求 404 或 500

  • 检查 app.register 的 prefix
  • 检查请求方法 GET/POST
  • 看后端终端错误日志
