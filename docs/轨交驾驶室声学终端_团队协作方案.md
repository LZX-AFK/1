### 1.1 口呼动作清单（功能①的输入）

地铁司机每站必须完成的标准化口呼动作：

| 阶段 | 动作代码 | 口呼内容 | 触发时机 |
|:---|:---|:---|:---|
| 发车前 | `CALL_DEPART` | "发车信号确认，准备发车" | 发车指示灯亮 |
| 出站 | `CALL_SPEED_UP` | "列车已提速，注意前方" | 离站后加速阶段 |
| 区间行驶 | `CALL_SPEED_LIMIT` | "限速XX，速度XX，确认" | 每区间至少1次 |
| 进站前 | `CALL_SIGNAL` | "信号确认，绿灯通过" | 进站信号机前 |
| 进站中 | `CALL_SPEED_DOWN` | "列车已降速，准备进站" | 减速进站阶段 |
| 停稳后 | `CALL_DOOR` | "屏蔽门安全确认" | 列车停稳后 |
| 发车前 | `CALL_DEPART_READY` | "站台安全，准备发车" | 确认站台安全后 |
| 紧急 | `CALL_EMERGENCY` | "紧急制动！" | 突发情况 |

> Demo 取其中 5-6 个核心动作。动作清单通过 JSON 配置文件定义，可扩展。

### 现有路由注册方式（在 `app.js` 中）

```js
// 每个路由文件导出 async function，接收 app 实例
export async function audioRoutes(app) {
  app.post('/upload', async (request, reply) => { ... })
}

// 在 app.js 中注册，prefix 决定完整路径
await app.register(audioRoutes, { prefix: '/api/audio' })
// 最终路径 → POST /api/audio/upload
```

**你们三人要做的，就是各写一个 `src/routes/xxx.js` 文件，然后在 `app.js` 里用同样的方式注册。**

### 3.2 Person A — 口呼监控模块

**核心目标：** 接收前端上传的录音 → 匹配口呼类型 → 检查是否漏呼 → 返回结果。

**步骤1：新建路由文件**

```
src/routes/callout.js
```

**步骤2：实现以下接口**

| 方法 | 路径 | 功能 | 请求示例 |
|:---|:---|:---|:---|
| `GET` | `/defs` | 获取口呼动作定义列表 | — |
| `POST` | `/defs` | 新增/更新口呼动作定义 | `{"action_code":"CALL_SIGNAL","label":"信号确认"}` |
| `POST` | `/upload` | 上传录音 + 自动匹配口呼类型 | multipart: `audio` + formData: `driver_id, station_id, train_no` |
| `GET` | `/check/:driverId` | 查询某司机当前行程的合规状态 | — |
| `GET` | `/stats` | 合规率统计（按司机/日期/线路） | `?driver_id=xxx&date=2026-05-28` |

**步骤3：核心逻辑 — 口呼匹配算法**

```
录音文件上传后：
  1. 先调 B 的转写接口，拿到文字（"信号确认，绿灯通过"）
  2. 跟口呼动作定义模板做关键词匹配
     "信号确认" → CALL_SIGNAL
     "限速"     → CALL_SPEED_LIMIT
     "屏蔽门"   → CALL_DOOR
     "发车"     → CALL_DEPART
  3. 记录到 callout_checks：
     { driver_id, station_id, callout_code, status: 'completed', recording_id }
  4. 对比该站点应做的所有口呼动作 → 发现漏呼 → status: 'missed'
  5. 返回：{ matched: 'CALL_SIGNAL', missed: ['CALL_DOOR'], alert: '请确认屏蔽门安全' }
```

**步骤4：存储扩展 — 新建**

```
src/storage/callout-store.js    ← 口呼定义/检查结果的JSON读写
data/callout-defs.json          ← 口呼动作定义（8条初始数据）
data/callout-checks.json        ← 每次口呼检查记录
```

**关键难点：**
- 口呼匹配准确率（噪音环境下转写可能不准 → 用关键词匹配兜底）
- 多司机并发上传 → JSON文件读写可能冲突 → MVP用内存缓存方案

### 3.3 Person B — 实时转写模块

**核心目标：** 收到录音文件 → 调用真实ASR引擎 → 返回转写文字 → 存库 → 前端可检索。

**步骤1：新建路由文件**

```
src/routes/transcribe.js
```

**步骤2：实现以下接口**

| 方法 | 路径 | 功能 | 请求示例 |
|:---|:---|:---|:---|
| `POST` | `/task` | 创建转写任务（上传音频，返回taskId） | multipart: `audio` + formData: `driver_id` |
| `GET` | `/result/:recordingId` | 查询转写结果 | — |
| `GET` | `/search` | 全文关键词搜索转写文本 | `?keyword=紧急制动&start=2026-05-01&end=2026-05-28` |
| `POST` | `/text` | 直接提交文字（给A的接口用） | `{"recording_id":"xxx","content":"信号确认"}` |

**步骤3：ASR引擎选择（三选一）**

| 引擎 | 优点 | 缺点 | Demo推荐 |
|:---|:---|:---|:---|
| 阿里云ASR | 中文识别最好，有免费额度 | 需注册+实名 | ⭐⭐⭐ |
| 讯飞语音 | 轨交场景可定制 | 商业授权贵 | ⭐⭐ |
| Whisper(本地) | 免费，离线可用 | 中文准确率略低，需GPU | ⭐ (备用) |

**推荐：阿里云一句话识别API（最简单，30秒以内音频，5万次免费额度）**

```
POST https://nls-gateway-cn-shanghai.aliyuncs.com/stream/v1/asr
Headers: X-NLS-Token: <临时token>
Body: 二进制PCM音频（16kHz, 16bit, 单声道）
Response: {"result":"信号确认绿灯通过","status":20000000}
```

**步骤4：存储扩展 — 新建**

```
src/storage/transcription-store.js   ← 转写结果的JSON读写
data/transcriptions.json             ← 转写记录
```

**步骤5：WebSocket实时推送（可选，加分项）**

```js
// 在 app.js 中加 WebSocket 支持（Fastify 原生 @fastify/websocket）
// B 的路由中：
app.get('/ws', { websocket: true }, (socket, req) => {
  // 转写完成后 → socket.send(JSON.stringify({ recording_id, content }))
})
```

前端可以实时看到"司机刚说完话 → 文字立刻跳出来"的效果。

**关键难点：**
- ASR API 调用延迟（3-8秒）→ 前端要有 loading 状态
- 网络不稳定 → 加重试机制 + 本地缓存
- 音频格式转换（前端可能是MP3，ASR要求PCM）→ 用 ffmpeg 或 Node.js 转码

### 3.4 Person C — 回放管理模块

**核心目标：** 管理员可以按多条件检索录音 → 在线播放 → 看到对应转写文字。

**步骤1：新建路由文件**

```
src/routes/playback.js
```

**步骤2：实现以下接口**

| 方法 | 路径 | 功能 | 请求示例 |
|:---|:---|:---|:---|
| `GET` | `/list` | 录音列表（分页+多条件筛选） | `?driver_id=xxx&start=2026-05-01&end=2026-05-28&callout_code=CALL_SIGNAL&page=1&size=20` |
| `GET` | `/detail/:recordingId` | 单条录音详情（含转写+口呼匹配） | — |
| `GET` | `/stream/:recordingId` | 音频流（支持Range，可拖动进度） | — |
| `GET` | `/timeline` | 时间轴聚合（某司机某天的全部事件） | `?driver_id=xxx&date=2026-05-28` |
| `GET` | `/export` | 打包导出证据链 | `?ids=id1,id2,id3` 返回ZIP |

**步骤3：音频流播放（Range请求支持）**

```js
// 关键：浏览器 <audio> 标签拖动进度条会发 Range 请求
// Fastify 原生支持，但需要手动处理

app.get('/stream/:recordingId', async (request, reply) => {
  const recording = await getRecording(request.params.recordingId)
  const filePath = recording.file_path  // 如 uploads/audio/xxx.wav
  const stat = await fs.stat(filePath)

  // 设置正确的 Content-Type 和 Accept-Ranges
  reply.header('Accept-Ranges', 'bytes')
  reply.header('Content-Type', 'audio/wav')
  reply.header('Content-Length', stat.size)

  // 如果前端发了 Range 头，按分段返回
  const range = request.headers.range
  if (range) {
    const parts = range.replace(/bytes=/, '').split('-')
    const start = parseInt(parts[0], 10)
    const end = parts[1] ? parseInt(parts[1], 10) : stat.size - 1
    reply.code(206)
    reply.header('Content-Range', `bytes ${start}-${end}/${stat.size}`)
    reply.header('Content-Length', end - start + 1)
    return fs.createReadStream(filePath, { start, end })
  }

  return fs.createReadStream(filePath)
})
```

**步骤4：时间轴聚合逻辑**

```
输入：driver_id + 日期
输出：按时间排列的全部事件

[
  { time: "08:00:05", type: "callout", callout: "CALL_DEPART", text: "发车信号确认...", recording_id: "xxx" },
  { time: "08:02:30", type: "callout", callout: "CALL_SPEED_UP", text: "列车已提速...", recording_id: "xxx" },
  { time: "08:02:35", type: "transcription", text: "限速60速度58确认", recording_id: "xxx" },
  { time: "08:05:00", type: "miss", callout: "CALL_DOOR", alert: "漏呼：屏蔽门安全确认" },
  ...
]
```

从 recordings 表（A的产出）、transcriptions 表（B的产出）、callout_checks 表（A的产出）三个数据源聚合。

**步骤5：存储扩展 — 只读已有数据**

C 不创建新的存储文件，而是**读取 A 和 B 产出的数据文件**：
- `data/records.json` — 现有
- `data/callout-defs.json` — A 产出
- `data/callout-checks.json` — A 产出
- `data/transcriptions.json` — B 产出

**C 的核心价值不是"存数据"，而是"把不同来源的数据串在一起呈现"。**

## 六、风险预案

| 风险 | 概率 | 预案 |
|:---|:---|:---|
| **阿里云ASR注册/审批慢** | 中 | 提前注册。备选：B 改用 `mock-ai.js` 模式返回预设转写文字 |
| **口呼匹配准确率低**（转写不完美） | 中 | 降级：不做语义匹配，只做关键词匹配（"信号"→CALL_SIGNAL，"限速"→CALL_SPEED_LIMIT） |
| **三人改 app.js 冲突** | 中 | 约定每人只加一行，改之前群里通知。合代码时手动处理。 |
| **C 的音频流在真机上播不了** | 低 | 提前用手机浏览器测试 `GET /api/playback/stream/:id`，确认 Content-Type 正确 |
| **现场网络不稳定** | 高 | 后端跑在笔记本 localhost，手机连笔记本热点，不依赖场馆 WiFi |
| **ASR API 当场挂掉** | 低 | B 提前准备 Mock 数据：用固定的转写文字代替真实API结果，确保Demo至少能"演"下去 |
